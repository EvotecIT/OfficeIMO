# OfficeIMO.Word Capability And Compatibility Matrix

This document tracks where `OfficeIMO.Word` is already strong, where support is partial, and where explicit feature work is still needed for mature Word document automation workflows.

It is intentionally honest. "Partial" means usable, not "done".

## Current Strengths

- ergonomic code-first `.docx` generation
- first-party dependency-free import, native subset write, and guarded conversion for supported legacy `.doc` files
- broad practical coverage for paragraphs, tables, sections, headers, footers, images, and content controls
- managed Open XML automation without requiring Office installation
- shared package security and capability preflight for safer automation against unknown documents
- byte-level macro and embedded-payload management without executing active content
- batch PNG/SVG page preview APIs with focused fixed-layout preflight
- adjacent HTML, Markdown, and PDF packages for destination-specific workflows
- bounded field and index refresh diagnostics, including two-column index concordance marking, imported `INDEX \c` column-count readback, and regenerated switch preservation while exact multi-column index layout remains partial

## Near-Term Market Readiness Focus

Word-to-PDF fidelity is tracked separately in `OfficeIMO.Word.Pdf` / `OfficeIMO.Pdf`. The non-PDF Word push should focus on the surfaces that can make `OfficeIMO.Word` stand out as a practical open-source Word automation engine:

1. Template and document assembly polish: make the existing merge fields, repeated regions, content-control bindings, validation, and batch output story obvious through scenario docs, proof artifacts, and workflow APIs.
2. Review, redline, and diff workflows: turn comments, revisions, visible markup, and `WordDocumentComparer` into a structured review story with deterministic diff output, review reports, and redline documents.
3. HTML and Markdown conversion fidelity: keep expanding the real-world conversion corpus, diagnostics, support matrix, and generated artifact gallery for Word/HTML/Markdown workflows.
4. Real-document docs and showcase: lead with concrete documents, source inputs, validation status, and known limitations instead of only listing object-model features.

Keep this section aligned with the generated proof-gallery example and the public Word market-readiness docs as implementation slices land.

The focused implementation plan for the remaining non-PDF, non-legacy-DOC gaps is tracked in [`Docs/officeimo.word-premium-gap-plan.md`](../Docs/officeimo.word-premium-gap-plan.md).

Expand digital signature support beyond structural preflight and the Windows package-signing adapter: broader external signed-document corpus validation, cryptographic validation, transform-aware OPC digest validation, certificate-chain trust, revocation/timestamp authority policy, macro-project signing, and cross-platform package signing remain roadmap items.
3. Expand review workflows beyond generated/imported-shape proof: broader Word-authored imported review corpus coverage beyond the initial sanitized review fixtures and full in-place visual redline generation.
4. Grow the identity-locked Word binary corpus beyond the current Microsoft Word COM fixture while retaining the broad generated DOC parser/writer contracts and preservation-sensitive Open XML fixtures.
5. Keep the interoperability guide and capability matrix aligned as file, stream, encrypted, macro-enabled, legacy binary, read-only, and unknown-document workflows evolve.
6. Keep PDF/layout validation out of release promises until the PDF stack is ready for that responsibility.
