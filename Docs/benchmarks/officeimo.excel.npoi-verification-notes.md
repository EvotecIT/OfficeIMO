# OfficeIMO.Excel NPOI Verification Notes

Date: 2026-06-25

This note documents the opt-in NPOI benchmark runner. It is not a product
roadmap, not a dependency proposal, and not an API design source. OfficeIMO
keeps its own agenda: dependency-free OpenXMLSDK-based APIs, high-level workbook
automation, explicit diagnostics, corpus evidence, and native OfficeIMO
projection paths.

The runner is useful only as one external reference implementation for checking
that benchmark fixtures exercise expected workbook signals. The verification
runner should answer narrow questions such as:

- do both readers see the same scalar `.xls` cell values in the generated fixture?
- do both readers expose equivalent formula text and cached value signals?
- do both readers see expected comments, hyperlinks, merged ranges, validations,
  conditional-formatting rules, AutoFilter range metadata, styles, and pictures?

It should not answer broader product questions such as which object model
OfficeIMO should expose, whether OfficeIMO should add a runtime dependency, or
whether OfficeIMO should pursue full binary `.xls` authoring before the import
model is mature.

## Boundaries

- NPOI stays in the opt-in `OfficeIMO.Excel.Benchmarks.NPOI` project.
- The project is intentionally outside `OfficeIMO.sln`.
- OfficeIMO runtime projects must not reference NPOI.
- The runner may accept benchmark-only package terms when needed for local
  evidence, but that acceptance must not leak into product/runtime projects.
- The checked-in result JSON is evidence for the measured fixture signals only.
- Different metrics in rich metadata lanes are not failure by themselves; the
  two libraries expose some workbook concepts through different object shapes.

## Current Lanes

The current opt-in runner covers scenarios that are natural benchmark checks:

- `xlsx-write-cellvalues`: plain row/cell writes to `.xlsx`.
- `xlsx-read-cellvalues`: plain row/cell reads from the same `.xlsx` shape.
- `xls-read-cellvalues`: scalar values from a generated `.xls` fixture.
- `xls-read-formulas`: formula text and cached values from a generated `.xls`
  fixture.
- `xls-read-metadata`: comments, hyperlinks, merged ranges, and list data
  validation signals.
- `xls-read-conditional-formatting`: simple cell-is and formula conditional
  formatting rule signals.
- `xls-read-autofilter-range`: AutoFilter range/drop-down metadata.
- `xls-read-styles`: font, fill, border, number-format, and alignment signals.
- `xls-read-pictures`: embedded PNG picture signals.

Do not add OfficeIMO-specific template, preflight, PDF, package-copy,
direct-package, or report-workflow scenarios unless there is an equivalent
benchmark surface that measures the same work without inventing artificial
adapter behavior.

## Evidence

The current checked-in result is:

`Docs/benchmarks/officeimo.excel.npoi-comparison-current.json`

It was generated by the opt-in runner with 2,500 rows, one warmup, and three
measured iterations. Equal metrics in scalar read, formula, AutoFilter-range,
and style-signal lanes are useful confirmation that both paths read the same
fixture signals. Rich metadata and picture lanes validate counts/ranges and
record coverage without requiring identical metric values.

SkiaSharp is referenced by the benchmark project only because the external
reader loads it for some HSSF drawing/comment paths. That reference must stay
local to the benchmark project.

## Direction

Continue the OfficeIMO legacy XLS work on its own architecture:

- parse BIFF records into a first-party legacy workbook model;
- project only content that can be mapped safely into normal OfficeIMO/Open XML
  APIs;
- report preserve-only content with stable diagnostics and corpus baselines;
- grow coverage from fixtures, corpus evidence, and user workflows;
- keep native `.xls` write/edit out of scope until the read/import model is
  stable enough to justify a separate design.
