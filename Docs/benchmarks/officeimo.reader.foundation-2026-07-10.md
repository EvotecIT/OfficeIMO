# OfficeIMO.Reader foundation benchmark

This is the first Reader-wide baseline after the instance-host, bounded async, content-detection, diagnostics, and stable transport foundation. It is evidence for regression work, not a cross-machine performance claim or service-level objective.

## Environment and method

- Apple M4, 10 physical/logical cores
- macOS 26.5
- .NET SDK 10.0.102
- .NET 8.0.23 Arm64 runtime
- BenchmarkDotNet 0.15.8 `ShortRun`: one launch, three warmups, three measured iterations
- deterministic corpus generated by `OfficeIMO.Reader.Benchmarks`; corpus creation is outside measured operations
- source hashing disabled in extraction benchmarks

Command:

```powershell
dotnet run --project OfficeIMO.Reader.Benchmarks/OfficeIMO.Reader.Benchmarks.csproj -c Release -f net8.0
```

All 25 benchmark cases completed: 14 rich document reads, five content-detection lanes, four Markdown parser/chunker lanes, and two schema transport lanes.

The ZIP corpus contains 20 table-heavy Markdown entries. Its allocation is approximately the per-entry Markdown cost multiplied by the entry count, which shows linear adapter composition but remains a material host-level budget.

## Markdown defect and fix

The first isolated run showed `MarkdownReader.ParseWithSyntaxTree` at about 73.7 ms and 237 MB for 80 sections containing 80 five-row tables. CPU/allocation tracing found that every parser `MarkdownDoc.Add(...)` rebound the entire growing semantic tree.

Parser-owned binding is now deferred until the final syntax tree exists. Public `MarkdownDoc.Add(...)` still binds immediately. On the same corpus:

| Lane | Before | After |
| --- | ---: | ---: |
| Syntax tree with tables | ~73.7 ms / 237 MB | 21.447 ms / 41.73 MB |
| Syntax tree without tables | 4.350 ms / 14.25 MB | 1.999 ms / 4.86 MB |
| Reader heading/table chunks | ~73.9 ms / 238.6 MB | 22.371 ms / 43.45 MB |
| Reader paragraph chunks | ~24.9 μs / 306.72 KB | 26.0 μs / 306.72 KB |

The change is protected by a table-heavy allocation contract and full semantic parent/sibling validation.

## Detection and transport

Bounded detection completed between 2.0 μs and 151.9 μs for the Markdown, PDF, Word, EPUB, and Visio inputs, allocating 64.9-154.4 KB per call.

Schema version 5 transport measured 977.9 μs / 877.69 KB to serialize and 2.022 ms / 1.10 MB to deserialize the representative Markdown rich result.

## Hierarchical chunking

The token-aware hierarchy lane uses the same deterministic 80-section Markdown result, a 512-token chunk cap, 64-token overlap, hierarchy context, and the built-in heuristic token counter. A focused short run measured:

| Lane | Mean | Allocated |
| --- | ---: | ---: |
| Build chunk hierarchy | 286.2 μs | 1.10 MB |
| Serialize hierarchy JSON | 1.174 ms | 969.53 KB |

The hierarchy builder run includes deterministic SHA-256 leaf IDs and hashes, source-span evidence, context accounting, and flattened document/heading nodes. These values establish regression evidence for the first implementation; they are not tokenizer-independent throughput claims. Exact host tokenizers should be benchmarked through their `IReaderTokenCounter` implementation.

## Next performance targets

The evidence identifies three owner-level allocation priorities:

1. HTML parsing and projection: 208.56 MB for the current 160-section corpus.
2. RTF parsing and projection: 110.33 MB for the current 300-section corpus.
3. ZIP aggregate budgeting: 855.65 MB for 20 table-heavy Markdown entries, even after the per-entry Markdown fix.

These should be optimized in the owning format/parser packages or through explicit aggregate budgets. The Reader facade should continue to provide routing, limits, and diagnostics without duplicating parser logic.
