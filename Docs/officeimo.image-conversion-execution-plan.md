## Ownership Contract

OfficeIMO image conversion has one reusable rendering owner:

```text
Document package
  -> document-specific visual snapshot and source policy
  -> OfficeIMO.Drawing primitives, raster canvas, SVG/PNG helpers, diagnostics
  -> thin document-specific public APIs
```

`OfficeIMO.Drawing` owns reusable pixels, paths, fills, strokes, clipping, text layout, image projection, chart/drawing primitives, PNG/SVG helpers, shared image-export result/options, and document-agnostic visual diagnostics.

Document packages own source semantics:

- `OfficeIMO.Excel`: workbook, worksheet, range, cell style, number format, conditional formatting, anchors, print/page policy.
- `OfficeIMO.PowerPoint`: slides, masters/layout inheritance, shapes, placeholders, fixed canvas, slide background, animation diagnostics when needed.
- `OfficeIMO.Word`: pagination, sections, paragraphs, tables, floating objects, headers/footers, page policy.
- `OfficeIMO.Visio`: VSDX page, shapes, connectors, stencils, routing, coordinate policy.
- `OfficeIMO.Pdf`: PDF stream/page writer behavior, but shared image-like geometry and PNG helpers come from Drawing.

Do not add private PNG encoders, raster buffers, SVG primitive writers, text wrapping engines, image-transform loops, or third-party raster dependencies in document packages.

## Execution Order

1. Keep the Excel-to-image base branch fresh in this worktree.
2. Preserve the public Excel range/sheet/workbook PNG/SVG API unless a real blocker appears.
3. Keep shared export result/options in `OfficeIMO.Drawing`.
4. Keep diagnostics stable and source-referenced.
5. Keep architecture guards passing before and after every shared-rendering slice.
6. Burn down Excel fidelity in narrow slices: charts, conditional formatting, text, objects/images, worksheet page export.
7. Move generic geometry, text, image, fill, stroke, chart, and QA behavior into Drawing as soon as a second consumer exists or is clearly imminent.
8. Start PowerPoint-to-image before Word-to-image because slides are fixed-size canvases.
9. Build PowerPoint image export as `PowerPointSlideVisualSnapshot -> OfficeIMO.Drawing -> PNG/SVG`.
10. Reuse PowerPoint PDF/background/shape extraction where it expresses PowerPoint source semantics, but do not turn PDF writer code into the image renderer.
11. Add PowerPoint visual baselines for text, images, shapes, backgrounds, tables, charts, and inherited layout.
12. Start Word-to-image only after the shared Drawing path and PowerPoint slide path are stable.
13. Build Word image export as page previews over Word layout semantics, not as an Excel-style range renderer.
14. Scope Word MVP to deterministic page previews with diagnostics for unsupported pagination/layout features.
15. Add shared visual-baseline helpers only when they prove real cross-document output.
16. Keep file growth under control; split by responsibility before adding substantial behavior to files already near or above the structure thresholds.
17. Prefer real artifact tests and visual baselines for rendering behavior.
18. Use focused unit tests for pure shared geometry, text, parser, and option contracts.
19. Treat red downstream checks from unpublished shared fixes as release-state evidence, not a reason for compatibility branches.
20. Keep every public surface thin: validate/map options, call the owning snapshot/renderer, return `OfficeImageExportResult`.
