# OfficeIMO.Excel Capability And Compatibility Matrix

This document tracks where `OfficeIMO.Excel` is already strong, where support is partial, and where explicit feature work is still needed for mature spreadsheet automation workflows.

It is intentionally honest. "Partial" means usable, not "done".

## Current Matrix

| Area | Status | Notes |
| --- | --- | --- |
| Workbook create/load/save | Supported | File and stream workflows are available, including normalized recovery for some malformed content-type packages. |
| Legacy XLS read/write | Partial | BIFF8 `.xls` files load through `ExcelDocument.Load(...)` with supported worksheet content projected into the normal OfficeIMO model and preserve-only legacy features reported through diagnostics. Password-to-open BIFF8 workbooks using XOR obfuscation, classic RC4, or RC4 CryptoAPI encryption can be imported through `ExcelDocument.LoadEncrypted(...)` or `LegacyXlsImportOptions.Password`. Legacy chart sheets are projected as Open XML chart-sheet package parts for `.xlsx` save/convert paths, with decoded chart metadata exposed while full visual chart reconstruction remains a chart-fidelity gap. Default and embedded BIFF theme records are projected into the normal workbook theme part. Native `.xls` save is available for the supported first-party BIFF8 writer subset through normal `Save("*.xls")` and stream save routes. `ExcelDocument.Convert(...)` provides a guarded file-to-file `.xls`/`.xlsx` helper over those same load/save paths. Unsupported legacy shapes such as remaining unsupported encryption variants, drawings, fully reconstructed chart visuals, PivotTables, worksheet table definitions, workbook connections/query tables, VBA/OLE content, and digital signatures remain diagnosed or preflight-blocked instead of being silently dropped. |
| XLSB read/write | Partial | `.xlsb` packages are detected by content and load through `ExcelDocument.Load(...)`. The BIFF12 importer projects supported values, shared strings, formulas and cached values, calculation settings, styles, dates, worksheet geometry, views, merges, defined names, and hyperlinks into the normal model while reporting preserved records. Unchanged sources copy byte-for-byte; supported cell edits use a native preservation-aware rewrite; and new workbooks can be written through the first-party XLSB writer. `ExcelDocument.Convert(...)` supports XLSB-to-XLSX conversion. Unsupported mutations and save-time transforms fail before output is written, so XLSX bytes are never mislabeled as XLSB. |
| XLSX/XLS/XLSB capability reporting | Supported | `ExcelFormatCapabilityReport.Current` exposes a stable, machine-readable matrix and Markdown output for native authoring, native subsets, compatible recipes, preservation-only behavior, and unsupported target-format combinations. It keeps parity decisions explicit instead of treating successful save as proof that every feature survived. |
| Typed range reads | Supported | `RowsAs<T>()`, `RowsAsStream<T>()`, `ReadObjects<T>()`, `ReadObjectsStream<T>()`, `ReadRangeAs<T>()`, and dictionary/DataTable reads are available, with friendly header-to-property matching, explicit aliases via `DisplayName`, `DataMember(Name=...)`, and `ExcelColumn`, plus diagnostics for ambiguous typed mappings. |
| Editable row workflows | Supported | `RowsObjects()` and read bridges support practical read-modify-save flows. |
| Header handling | Supported | Duplicate, normalized-colliding, blank, and generated-fallback-vs-explicit headers now disambiguate deterministically, and header-map lookups stay fresh after in-memory edits. |
| Number/date import fidelity | Partial | Common formats work well; custom formats are better after the token-aware classifier, but the corpus still needs to grow. |
| Tables and named ranges | Supported | Table creation and naming safeguards are in place, with worksheet/global named range helpers. |
| Auto-fit and report ergonomics | Supported | Auto-fit, object insertion, and table/report helpers are a current strength. |
| Image and report rendering | Partial | Range, worksheet, and workbook PNG/SVG export supports page areas, print areas, manual page-break slicing, headers/footers, comments, images, charts, shapes, conditional formatting, sparklines, and stable approximation diagnostics through shared Drawing primitives. Approved visual baselines and a dedicated CI smoke gate protect representative output. It is not Microsoft Excel's layout engine; unsupported or approximate features remain explicit in export diagnostics. |
| Charts | Partial | Common chart authoring is present, including dashboard recipe helpers, KPI/contribution/variance-bridge presets, stacked/100% stacked column/bar/line/area variants, 3-D area/line/column/bar/pie, pie-of-pie/bar-of-pie, radar, stock, and filled/wireframe/contour surface charts, axis titles/formats/scale/gridlines, series styling, trendlines, secondary-axis combo scenarios, and pivot-source chart metadata, but breadth, Excel-authored mutation, and round-trip coverage still need work. |
| Modern chart workflows | Partial | `AddHistogramChart(...)`, `AddParetoChart(...)`, `AddFunnelChart(...)`, and `AddWaterfallChart(...)` calculate bins, cumulative percentages, spacers, and bridge series and write compatible XLSX charts with inspectable cached data. These are intentionally classic-chart recipes, not mislabeled ChartEx parts. Native ChartEx authoring and binary XLS/XLSB chart creation remain open. |
| Pivot tables | Partial | Source-range pivot creation supports row/column/page/data fields, layouts, styles, display flags, captions, data-field number formats with built-in/custom id/code readback, show-values-as data field options, field sort/subtotal/display/layout/number-format options with composable fluent helpers and built-in/custom id/code readback, field item visibility filters with hidden/visible item readback, selected page-field items with readback, fluent item/page filter helpers, label/value pivot filters with common comparison and negated variants, fixed and dynamic date filters, top/bottom count/percent/sum filters, formula-backed calculated fields with number-format id/code readback, date/number grouping metadata, generated multi-level date hierarchy fields with base/parent relationships, and explicit grouped-cache item metadata. Feature inspection now flags slicer, timeline, connection, and query-table package metadata as preserve-only before edit-heavy workflows, with round-trip preservation coverage for those package parts. Deeper Excel interoperability checks, calculated items/members, slicer/timeline authoring, connection/query authoring, and broader advanced filters remain open targets. |
| Pivot source updates | Partial | `UpdatePivotTableSource(...)` validates the new header contract, detects shared caches, updates worksheet source metadata, invalidates stale cache records, and requests refresh on open. It currently targets worksheet-backed XLSX pivots; native BIFF8/BIFF12 pivot rewrites remain open. |
| Pivot slicer/timeline bindings | Partial | `AddPivotSlicerCache(...)` and `AddPivotTimelineCache(...)` validate the pivot table and source field before writing OfficeIMO-owned binding metadata, and `GetWorkbookSlicerCaches()` / `GetWorkbookTimelineCaches()` provide typed readback. The metadata uses OfficeIMO relationship/content types so Excel can safely ignore it; native Excel cache structures and UI shapes are not materialized yet. |

A1-backed named range references, simple table structured references, common text helpers including `CONCATENATE`, `TEXTJOIN`, `TEXTBEFORE`, `TEXTAFTER`, `FORMULATEXT`, `SUBSTITUTE`, `FIND`, `SEARCH`, `COVAR`, `SUMSQ`, `SUMPRODUCT`, `STDEV.S`, `STDEV.P`, `VAR.S`, `VAR.P`, `PERCENTILE.INC`, `PERCENTILE.EXC`, `QUARTILE.INC`, `QUARTILE.EXC`, `PERCENTRANK.INC`, `PERCENTRANK.EXC`, `RANK.EQ`, `RANK.AVG`, `CORREL`, `SLOPE`, `INTERCEPT`, `RSQ`, and `FORECAST.LINEAR`, bounded financial report formulas including `PMT`, `PV`, `FV`, `NPER`, and `NPV`, report rounding helpers including `ROUND`, `ROUNDUP`, `EDATE`, `EOMONTH`, `DAYS`, `DAYS360`, `WEEKDAY`, `WEEKNUM`, `ISOWEEKNUM`, `NETWORKDAYS`, `WORKDAY`, and `WORKDAY.INTL`, conditional aggregations with literal/range/cell criteria including `MINIFS` Broader formula compatibility remains a target. |
| Formula depth and circular references | Supported | `MaximumDependencyDepth` bounds nested evaluator work and leaves over-budget formulas intact for Excel. Dependency inspection resolves direct A1 references, defined names, and supported structured table references before reporting formula-to-formula edges, longest resolvable depth, dependents, and strongly connected circular-reference groups without recursive graph traversal. The shared diagnostics and tested formula subset round-trip through XLSX, XLS, and XLSB. |
| Custom formula functions | Supported | `ExcelCalculationOptions.RegisterCustomFunction(...)` extends calculation with application-owned functions, typed values, worksheet/cell context, deterministic name validation, and explicit removal. Registered functions participate in formula inspection, dependency checks, direct `Calculate()`, and save-time cached-value evaluation while the original formula text remains intact. |
| Macros and embedded payloads | Partial | VBA projects can be inspected with optional SHA-256, attached, extracted, saved, and removed without execution. Embedded package, OLE, and ActiveX payloads can be inventoried, hashed, extracted with byte limits, replaced while keeping their owner relationship, and removed with known referencing markup. OfficeIMO does not execute or edit VBA source modules, sign VBA projects, or provide a complete OLE/ActiveX object model. |

Pre-generation inspection reports missing bindings plus bound value kinds/types for text, numbers, dates, durations, booleans, nulls, images, and objects. | Comments and notes | Partial | Legacy worksheet comments can be authored, inspected, found, updated, and removed by author, text, and A1 range, including rich-text comment runs for authored and updated comments. Threaded comments can be read, updated, resolved, reopened, and removed with workbook person metadata and conversation-id integrity checks. New threaded-conversation authoring and full Excel collaboration semantics remain partial. |
| Worksheet/workbook protection | Partial | Protection helpers exist, but broader permission fidelity and compatibility proof are still needed. |
| Encryption/password support | Supported | Password-encrypted Office Open XML workbook open/save is available through `LoadEncrypted`, `LoadEncryptedAsync`, and `SaveEncrypted`. Legacy `.xls` password-to-open import supports XOR-obfuscated, classic RC4, and RC4 CryptoAPI FilePass workbooks when a password is supplied. Worksheet/workbook protection remains a separate feature area. |
| Package security | Supported | `ExcelLoadOptions.PackageSecurity` applies shared package, part, aggregate-size, compression-ratio, unsafe-name, relationship, macro, embedded-payload, ActiveX, and external-relationship policy before Open XML, XLSB, or compound XLS parsing. `SecureDefaults` retains active-content compatibility with structural limits; `UntrustedDefaults` rejects active and external content. Typed findings and `OfficePackageSecurityException` expose the rejecting rule and part. Omitting the option retains historical compatibility behavior. |
| Streaming for very large workbooks | Partial | In-memory/file/stream workflows are strong, and a large-workbook guidance document now describes recommended generation, read, preflight, and benchmark paths. Broader row-count and hardware evidence still needs regular refreshes. |
| Import fidelity for ugly real-world workbooks | Partial | The executable corpus inventories 29 Excel binary artifacts: 22 positive BIFF8 files, two BIFF diagnostic files, and five Microsoft Excel-authored XLSB workbooks, with SHA-256 identity, provenance, focus, and approved legacy import reports. It covers Apache POI, Microsoft Excel COM, and OpenPreserve sources. Broader customer-like workbook diversity remains a target. |
| Feature inspection | Partial | `InspectFeatures()` reports editable, partially editable, preserved, and unsupported workbook features, with detail entries for preservation-sensitive package parts and external relationships, including slicer, timeline, connection, query-table, threaded comment, digital signature, OLE object, and form-control package signals. Capability preflight distinguishes data reads, cached-formula reads, calculation, cell edits, structural edits, template binding, and fixed-layout report export; `Can`, `EnsureCan`, capability diagnostics, repair hints, and Markdown reporting support deterministic routing. Preservation proof includes slicer/timeline, connection/query-table, threaded comment/workbook person, digital signature origin/app-property metadata, OLE/form-control package metadata, custom XML, external links, macros, and embedded package parts. |
| Public benchmark evidence | Partial | Committed benchmark snapshots plus write/read profiles now exist. Recent local snapshots show strong OfficeIMO results on covered write/read/load-edit-save workloads, including the refreshed 25,000-row report-export and direct DataSet export profiles, but broader row-count and hardware coverage is still needed before making large-workbook claims. |

## Current Strengths

- ergonomic write APIs for reports, tables, and object insertion
- deterministic save/repair behavior compared with raw Open XML usage
- typed and editable read surfaces that are pleasant for application code
- practical stream support for service and automation scenarios
- first-party BIFF8 `.xls` import, native subset write, and guarded `.xls`/`.xlsx` conversion
- first-party BIFF12 `.xlsb` import, native subset write, preservation-aware rewrite, and guarded XLSB-to-XLSX conversion
- shared package security, feature/capability preflight, and byte-level macro/embedded-payload management

## Highest-Priority Gaps

1. Keep reducing report-export overhead, with table creation, auto-fit, and save costs now as important as `InsertObjects()` on the 25,000-row profile.
2. Publish and refresh benchmark result sets on stable hardware instead of relying on a single developer machine snapshot.
3. Grow the identity-locked 29-file Excel binary corpus with more independent producers, locales, encryption variants, and complex business workbooks.
4. Keep extending formula/recalculation coverage while preserving a clear boundary between built-in evaluation, registered application functions, cached results, and formulas delegated to Excel.
5. Keep chart/pivot expansion driven by interoperability tests rather than isolated feature requests.

## Latest Snapshot Highlights

- A 2026-07-07 focused Release write scan kept the public API shape clean: normal `InsertDataTable`, `InsertDataReader`, `InsertObjects`, and direct `CellValues` paths were measured without benchmark-only tuning switches. At 10,000 rows, `OfficeIMO.Excel` averaged `19.2 ms` for DataTable write, `19.8 ms` for styled DataReader table write, `20.1 ms` for plain DataReader write, `17.8 ms` for direct CellValues rectangle write, and `16.9 ms` for typed object insertion. These lanes stayed far ahead of ClosedXML, EPPlus, and MiniExcel in the same local comparison while remaining close to the narrow LargeXlsx/Sylvan streaming lanes.
- The same 2026-07-07 DataReader investigation rejected internal changes that looked plausible but regressed timing or allocation, including bulk `DataTableReader.GetValues(...)`, disabling shared-string planning for buffered rows, and increasing the direct-save row-buffer starting capacity. Current evidence favors keeping the clean `InsertDataReader(...)` API and optimizing the owner engine only when focused A/B runs prove a real win.
- Updated 5-sample end-to-end snapshot on 2026-05-12: `OfficeIMO.Excel` averaged `114.9 ms` for the 2,500-row report scenario.
- The 2026-05-13 local Release comparison records build configuration, medians, raw samples, and deterministic read checksums. Current-library scenarios are measured in rotated groups so fixed library ordering does not decide the result. Covered read scenarios show strong bounded-read performance, while full dense materialization remains a priority target.
- The same 2026-05-13 Release comparison shows the strongest OfficeIMO shape on bounded reads: first-100-row range read averaged `3.8 ms`, and first-100-row streaming averaged `3.5 ms`.
- DataTable materialization is now covered explicitly: `ReadRangeAsDataTable()` averaged `58.1 ms` in the refreshed comparison. This is useful progress, while dense materialization shapes remain priority targets.
- Typed `ReadObjects<T>()` averaged `64.1 ms` in the refreshed comparison. A separate `ReadObjectsStream<T>()` path now exists for low-buffer typed row enumeration and is tracked by the `read-objects-stream` comparison scenario.
- Full 2,500-row dense reads are honest targets rather than solved claims: `ReadRange()` averaged `64.8 ms`, and `ReadRangeStream()` averaged `47.6 ms`.
- AutoFit on an existing workbook averaged `106.0 ms` in the refreshed comparison and remains a profile to keep watching as workbook metadata preservation grows.
- The latest 2,500-row write profile dated 2026-05-12 shows the current staged cost shape with report-export AutoFit saves deferred to the document boundary: `InsertObjects()` is about `16.8 ms`, `AddTable()` is about `27.8 ms`, `AutoFitColumns()` is about `25.4 ms`, and the OfficeIMO staged write total is about `99.4 ms`.
- A 25,000-row write profile on 2026-05-12 shows the larger-report shape after the row-major append, appended-cell style-cache, bulk shared-string registration, table range-scan, contiguous table-range verification, column-reference caching, auto-fit planning, auto-fit shared-string text/run caching, auto-fit style/number-format cache fast paths, and deferred AutoFit worksheet saves: `InsertObjects()` is about `211.4 ms`, `AddTable()` is about `153.1 ms`, `AutoFitColumns()` is about `169.0 ms`, and the OfficeIMO staged write total is about `720.7 ms`. The AutoFit profile records `BuildPlan`, `CalculateWidths`, and `ApplyWidths` sub-stages.
- A 2026-05-18 local Release comparison covers the automatic direct DataSet export path through normal `InsertDataSet(...)` plus normal `Save(...)`. At 25,000 rows, `OfficeIMO.Excel` averaged `42.9 ms` for styled DataSet tables, `45.6 ms` with AutoFit, and `35.2 ms` for sparse tables. This is a covered export-shaped workload, not a blanket claim for feature-rich edited workbooks.
- Fast package saves are automatic when the workbook is eligible. `ExcelSaveOptions.DisableFastPackageWriter` provides an explicit troubleshooting and validation opt-out; package-writer selection telemetry is internal so normal callers are not coupled to implementation paths.
- Formula calculation can now be invoked directly with `doc.Calculate()` or scoped to a single save with `ExcelSaveOptions.EvaluateFormulasBeforeSave`; supported same-sheet formula dependencies, numeric cross-sheet references, A1-backed named ranges, simple table structured references, common text helpers including join, before/after delimiter extraction, formula text, cleanup, search, exact comparison, repetition, conversion, and case-normalized labels, bounded `TEXT` number/date/time report labels, A-value aggregates over mixed report ranges, bounded statistical and financial report formulas including descriptive statistics, deviation, paired-array, rank, squared-sum, and sum-product summaries, standard/significance rounding helpers including negative digit positions for report-scale rounding, reference-shape helpers, report date/time parsing, week numbering, 360-day calendar, and workday scheduling functions, conditional aggregations with criteria cells including `MINIFS` and `MAXIFS`, bounded `COUNTBLANK` and `SUBTOTAL` report summaries, lookup formulas including `INDEX`, `MATCH`, `XMATCH`, and `XLOOKUP` bounded exact/next-smaller/next-larger modes, text-returning lookup results, value-returning `IF`/`IFS`/`SWITCH`/`CHOOSE`/`IFERROR`/`IFNA` report formulas, and `ISBLANK`/`ISNUMBER`/`ISTEXT`/`ISERROR`/`ISERR`/`ISNA`/`ISFORMULA` report guards are evaluated in-memory, while unsupported formulas stay intact, get clearer inspection diagnostics, workbook-level dependency graphs, and dependency issue guards, and can be paired with `ForceFullCalculationOnOpen` when Excel should finish the workbook calculation.
- The benchmark artifacts now include raw samples and medians so outliers are visible instead of hidden behind a single average.

## Suggested Release-Prep Checks

1. Build `OfficeIMO.Excel` and the benchmark harness on the target SDKs.
2. Run the Excel-focused test slice before any release candidate.
3. Run `./Build/Test-OfficeInteroperabilityGate.ps1 -Suite Full` to verify binary corpora, preservation, and safety contracts.
4. Run at least one benchmark class in `Release` to catch accidental performance cliffs.
5. Update this matrix whenever a major Excel feature is added or a documented gap is closed.
